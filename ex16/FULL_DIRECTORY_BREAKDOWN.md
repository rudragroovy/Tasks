# Species Pipeline Directory & File Breakdown

Here is a complete breakdown of every file and folder in the `species-pipeline` project, including hidden files, detailing what they do and why they exist:

## ⚙️ Core Configuration & Environment (Hidden & Root Files)
* **`.env`** – Holds your private runtime environment variables (like your `DATABASE_URL` and your API keys for OpenRouter and IUCN). This ensures keys are injected securely at runtime without hardcoding them in Python files.
* **`.env.example`** – A safe, dummy version of `.env` meant to be committed to version control. It shows other developers what keys they need to provide to run the project.
* **`.gitignore`** – Tells Git which files and folders to ignore (like `.env`, `__pycache__`, and your downloaded `cache/` data) so you don't accidentally push secrets or heavy cache files to GitHub.
* **`poetry.lock` & `pyproject.toml`** – These are the core Python dependency management files used by Poetry. `pyproject.toml` defines your project (name, version) and its dependencies (like `pydantic`, `sqlalchemy`, `openai`), while `poetry.lock` strictly pins the exact versions of those packages so the code runs exactly the same on any machine.
* **`docker-compose.yml` & `Dockerfile`** – `docker-compose.yml` spins up your local PostgreSQL database on port 5433 with a single command. The `Dockerfile` provides the build instructions if you ever want to containerize the Python pipeline itself.

## 🛠️ Python Development & Tooling Caches (Hidden Folders)
* **`.mypy_cache/`** – Hidden folder generated by `mypy`. It caches type-checking results to make subsequent runs of the static type checker much faster.
* **`.pytest_cache/`** – Hidden folder generated by `pytest`. It remembers which tests failed last time so you can re-run only the failed tests quickly.
* **`.ruff_cache/`** – Hidden folder generated by `ruff`. Caches the results of the Python linter/formatter to speed up linting.
* **`mypy.ini`, `pytest.ini`, `ruff.toml`** – The configuration files that tell those respective development tools exactly what rules to follow (e.g., telling `ruff` to enforce a certain line length).

## 📖 Documentation Files
* **`README.md`** – The main landing page for the project explaining what it is and how to run it.
* **`ARCHITECTURE.md`** – Explains the Hexagonal Architecture and asynchronous fan-out flow of the application.
* **`API_SOURCES.md`** – Documents the rules, rate limits, and quirks for the 5 APIs (GBIF, iNaturalist, Wikipedia, Wikidata, IUCN).
* **`LLM_POLICY.md`** – Documents the cost constraints, temperature settings, and rules to prevent hallucination for the LLM fallback.
* **`MERGING_RULES.md`** – Defines the conflict resolution logic (e.g., if Wikipedia and GBIF both return a description, whose is kept?).
* **`DIRECTORY_STRUCTURE.md`** – A high-level overview of the folder layout.
* **`PROMPTS.md`** – Tracks the system and user prompts specifically fed to OpenRouter/OpenAI.
* **`CONTRIBUTING.md`** – Explains how other developers should format code if they want to contribute to the repository.

## 🗄️ Database & Storage
* **`alembic.ini` & `migrations/`** – Alembic is the database migration tool for SQLAlchemy. It tracks the schema changes. The `alembic.ini` configures the connection, and the `migrations/versions/` folder contains the Python scripts that actually execute `CREATE TABLE` inside Postgres.
* **`cache/`** – A local folder created by `diskcache` to store raw API JSON responses. If the pipeline crashes, it can restart instantly by reading from this folder instead of re-hitting the network.

## 🚀 The Codebase (`app/` Directory)
The `app/` folder contains the actual logic, strictly separated using Hexagonal Architecture:
* **`cli/`** – Contains `main.py`. This is the entry point that uses `Typer` to handle the command-line arguments (like parsing `--input species.csv`).
* **`models/`** – Contains `domain.py`. This defines the strict Pydantic data schemas (like `SpeciesRecord`) that enforce data types across the entire app.
* **`resolver/`** – Handles canonical name resolution. Ensures that if you type a misspelled or outdated name, it resolves it to the accepted scientific name.
* **`adapters/`** – Contains the API fetchers (`gbif.py`, `iucn.py`, `wikipedia.py`, etc.). Each adapter is responsible for talking to a specific external API.
* **`interfaces/`** – Contains `adapter.py`. Defines the `BaseAdapter` blueprint that all the adapters must strictly inherit from.
* **`orchestrator/`** – Handles the `asyncio.gather()` fan-out. It takes a species name and fires off all 5 API adapters at the exact same time concurrently.
* **`merger/`** – Takes the 5 different JSON responses from the orchestrator and fuses them into a single `SpeciesRecord`, applying the `MERGING_RULES.md` conflict priorities.
* **`gap_detector/`** – Inspects the fused record, counts which fields are still empty, and creates a strict list of missing fields.
* **`llm/`** – Contains `client.py`. Takes the missing field list and sends a prompt to OpenRouter to predict the missing ecological data.
* **`validator/`** – Contains `validators.py`. A final security checkpoint that runs the Pydantic model and strips out hallucinations (like negative weights).
* **`storage/`** – Contains `database.py` and `upsert.py`. Takes the finalized validated record and safely saves it to the PostgreSQL database using `asyncpg`.
* **`logging/`** – Contains `logger.py`. Configures `structlog` to output highly readable, structured status messages in the terminal.
* **`caching/`** – Contains the `diskcache` wrapper that automatically intercepts network requests and caches them to the disk.
* **`reporter/`** – Contains `builders.py`. Calculates statistics on how many fields were filled and generates the final `coverage_report.md`.

## 🧪 Tests & Inputs
* **`species.csv`** – Your input dataset containing the 50 original species names to query.
* **`coverage_report.md`** – The output artifact generated at the very end of the pipeline detailing your success rates.
* **`check_apis.py`** – A manual script you can run to concurrently ping the 5 biodiversity APIs for a specific species to verify their JSON payloads and keys.
