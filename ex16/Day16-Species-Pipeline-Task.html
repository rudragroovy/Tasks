Species Data Aggregation Pipeline

Build a full pipeline that ingests species names, fans out across multiple free public APIs, merges results into a unified schema, fills gaps with an LLM, and stores a dense biodiversity dataset. Pure R&D · pure code · no single API will give you everything.

1. Objective

Input: list of species names (CSV or JSON). Output: normalized dataset of 50+ species with extensive per-species fields, full source provenance, and an LLM-fill report. Ship a working CLI pipeline, a stored dataset, and a coverage report.

No single source covers every field. The interesting work is in adapter design, schema merging, conflict resolution, gap detection, and disciplined LLM fallback.

2. Why R&D · what's being tested

  • Real data engineering · multiple flaky 3rd-party APIs · rate limits · partial responses.
  • Schema design · choose canonical fields · resolve conflicts between sources.
  • Disciplined LLM usage · only fill gaps · never override authoritative data · track provenance.
  • Code architecture · 1 adapter per source · single orchestrator · clean separation.
  • Cost discipline · async fan-out · response caching · prompt caching for LLM.

3. Free data sources · use ≥ 4

Pick at least 4. REQUIRED sources have no auth. OPTIONAL need a free token. Document why you chose each in sources.md.

GBIF required · no auth
https://api.gbif.org/v1/
Taxonomy backbone · name resolution (species/match) · occurrence counts · geographic distribution. Authoritative for canonical scientific names.
iNaturalist required · no auth
https://api.inaturalist.org/v1/
Common names · observation counts · user photos · regional presence. Strong for vernacular naming across languages.
Wikipedia REST required · no auth
https://en.wikipedia.org/api/rest_v1/page/summary/{title}
Plain-language description · thumbnail image · canonical article URL. Best for human-readable summary and quick image.
Wikidata SPARQL required · no auth
https://query.wikidata.org/sparql
Structured facts · lifespan · mass · length · diet type · taxon parent. Query by Wikidata QID resolved from Wikipedia.
EOL · Encyclopedia of Life required · no auth
https://eol.org/api/
Aggregated traits · habitat · behavior · multi-source attribution. Good fallback aggregator.
IUCN Red List optional free token
https://apiv3.iucnredlist.org/api/v3/
Conservation status · population trend · threat categories. Token via free email registration. Authoritative for conservation fields.
ITIS optional
https://www.itis.gov/ws_description.html
Cross-check taxonomic hierarchy. Useful when GBIF and Wikidata disagree.
NCBI Taxonomy optional
https://eutils.ncbi.nlm.nih.gov/entrez/eutils/
Scientific classification from a molecular-biology angle. Free, generous rate limit with a key.
Wikimedia Commons optional
https://commons.wikimedia.org/w/api.php
High-quality images via category traversal. Use to populate image_urls[] beyond Wikipedia thumbnail.
BHL · Biodiversity Heritage Library optional
https://www.biodiversitylibrary.org/api3/
Historical literature references. Stretch use only — adds depth for rare species.

5. Pipeline architecture

┌─────────────┐ │ species.csv │ scientific or common names └──────┬──────┘ ▼ ┌──────────────────────┐ │ Resolver │ GBIF species/match → canonical name + taxonKey └──────┬───────────────┘ ▼ ┌──────────────────────────────────────────────┐ │ Fan-out adapters (async / parallel) │ │ GBIF · iNat · Wikipedia · Wikidata · EOL │ │ IUCN · ITIS · NCBI · Commons │ │ each → { partial_record, source_tag } │ └──────┬───────────────────────────────────────┘ ▼ ┌──────────────────────┐ │ Merger │ per-field priority rules │ │ record field_provenance └──────┬───────────────┘ ▼ ┌──────────────────────┐ │ Gap detector │ list missing / low-confidence fields └──────┬───────────────┘ ▼ ┌──────────────────────┐ │ LLM filler │ JSON-only output · cite confidence │ (Claude / OpenRouter)│ never overwrites API data └──────┬───────────────┘ ▼ ┌──────────────────────┐ │ Validator │ type · range · sanity checks └──────┬───────────────┘ ▼ ┌──────────────────────┐ │ Storage │ Postgres (JSONB) or MongoDB │ │ idempotent upsert on scientific_name └──────┬───────────────┘ ▼ ┌──────────────────────┐ │ Reporter │ coverage_report.md └──────────────────────┘

Per-field priority rules · sample

taxonomy            → gbif > itis > ncbi > wikidata
conservation_status → iucn > wikidata > llm
description         → wikipedia > eol > llm
common_names        → inat > wikipedia > gbif
lifespan_years      → wikidata > eol > llm
image_urls          → commons > wikipedia > inat
behavior            → eol > wikipedia > llm
geographic_range    → gbif > iucn > wikidata

6. LLM guardrails · non-negotiable

Rules for LLM use.
  • LLM never overwrites a field already filled by an API.
  • LLM only fills missing fields listed by the gap detector.
  • Force JSON-only output via schema-bound prompt. Reject malformed responses.
  • Every LLM-filled field appears in llm_filled_fields[] with a confidence ∈ [0, 1].
  • Share system prompt across all species via prompt caching. Vary only the per-species facts.
  • Validator rejects LLM hallucinations (e.g., lifespan_years <= 0, weight in wrong units).
  • Total LLM spend for full 50-species run must stay under $2. Log token totals.

9. Constraints

Hard rules.
  • Free tier only. No paid API spend. IUCN token via free registration only.
  • Respect rate limits. Implement backoff and retry per source. No request hammering.
  • Pure code. No Zapier · n8n · Make · visual workflow tools.
  • Async I/O. Fan-out must be parallel (asyncio.gather · Promise.all).
  • Cache raw API responses. Disk cache prevents re-fetch during dev iteration.
  • LLM spend < $2 for the full 50-species run. Token log mandatory.
  • Idempotent. Re-running on same input must not duplicate or corrupt rows.

11. Evaluation rubric

CategoryWeightCriteria
Coverage 25% ≥ 80% non-null on core fields (taxonomy · description · conservation_status · habitat · image) across 50 species. Verified via coverage_report.md.
Provenance 15% Every field traceable to a source or explicitly tagged LLM. field_provenance populated for 100% of fields. llm_filled_fields[] accurate.
Resilience 15% Pipeline survives 1 dead source without crashing. Backoff + retry implemented. Failures logged with source + species + error.
Code quality 15% Modular adapters · shared interface · no copy-paste between source clients. Clear separation of resolver · merger · LLM · validator · storage.
Cost discipline 10% LLM spend < $2 for full run. Prompt caching wired. Raw API responses cached on disk. Token report present.
LLM discipline 10% LLM never overwrites API data. JSON-only output. Confidence scores attached. Validator catches at least 1 demonstrable hallucination.
Documentation 10% Fresh reader runs pipeline in < 10 min from README. Architecture diagram present. sources.md covers all chosen APIs.
